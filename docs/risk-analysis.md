## 一、模块功能概述

该模块实现了一个基于 TCP 协议的文件下载客户端：用户输入文件名，客户端连接本地服务器（127.0.0.1:34000），发送文件名请求，接收并写入文件。涉及网络通信、用户输入处理、文件系统写入三个核心操作，均属于高风险操作面。

---

## 二、敏感操作识别

| 操作类型 | 代码位置 | 说明 |
|----------|----------|------|
| 用户输入接收 | `main()` → `input()` | 用户直接输入文件名，无格式限制 |
| 文件名校验 | `validate_filename()` | 仅检查 `..`，防护不完整 |
| 文件系统写入 | `receive_file()` → `open()` | 以用户输入的文件名直接写入本地文件 |
| TCP 明文传输 | `connect_server()` → `socket` | 无 TLS/SSL 加密，数据明文传输 |
| 异常处理 | 多处 `except` | 部分异常捕获过宽，使用裸 `Exception` |

## 三、主要安全风险清单

### 风险 1：路径穿越漏洞（Path Traversal）⚠️ 高危

**描述：** 服务器返回的文件名或用户输入中若包含 `../../../etc/passwd` 等序列，当前仅检查了 `..` 但未规范化路径，导致写入目标超出预期目录。

**攻击场景：** 服务器端恶意构造响应报文，将文件名改为 `../../autorun.py`，客户端直接写入到上层目录。

**影响：** 任意文件写入，可覆盖系统关键文件。

**CVSS 概估：** 7.5（高）

---

### 风险 2：明文传输中间人攻击（MITM）⚠️ 中危

**描述：** 当前 TCP 连接无任何加密措施，所有传输内容（文件名请求、文件内容）均以明文传输。

**攻击场景：** 攻击者在局域网内使用 ARP 欺骗劫持连接，窃取传输文件内容或篡改响应数据注入恶意内容。

**影响：** 数据泄露、数据完整性被破坏。

**CVSS 概估：** 5.9（中）

---

### 风险 3：接收内容未校验（Malicious Content Injection）⚠️ 中危

**描述：** 客户端接收服务器返回的任意二进制内容并直接写盘，未对文件大小、MIME 类型或内容做任何限制。

**攻击场景：** 恶意服务器返回超大文件导致磁盘写满（拒绝服务），或返回可执行脚本伪装成普通文件。

**影响：** 磁盘资源耗尽、本地执行恶意代码。

**CVSS 概估：** 5.3（中）

---

### 风险 4：裸 Exception 捕获掩盖安全异常（次要）

**描述：** 多处使用 `except Exception` 捕获所有异常并仅打印日志，导致安全相关异常（如连接被重置、证书错误等）与普通异常混同处理，失去异常审计能力。

**影响：** 安全事件难以溯源。

---

### 安全风险分析报告

> **模块：** `成员代码/renyanbin/amia_defense_test.py`   
> **分析时间：** 2026-06-26  
> **分析人：** 谢智卓

经过对提供的自动化测试与防御框架代码进行审查，发现以下安全风险。按严重程度（高/中/低）排序，每个风险均包含漏洞类型、攻击场景描述、影响范围，并对已有但不完整的防护措施进行说明。

---

#### **高风险**

**1. 路径遍历漏洞（文件读取）**  
- **漏洞类型**：路径遍历（Path Traversal）  
- **攻击场景**：攻击者通过控制数据文件 `data/figstep_samples.jsonl` 中的 `"image"` 字段，构造类似 `"../../etc/passwd"` 的路径。程序未经验证直接调用 `Image.open(img_path)`，导致可读取系统任意文件（受当前进程权限限制）。  
- **影响范围**：泄露服务器敏感文件（如配置文件、密码文件、源代码等），若结合其他漏洞可进一步提权或横向移动。  
- **已有防护**：`try-except` 仅捕获图片加载异常（如格式错误），但未对路径合法性做任何检查（如限制目录、过滤 `..` 等）。防护不完整，无法阻止路径遍历。

**2. Python 模块搜索路径污染（代码执行）**  
- **漏洞类型**：模块注入 / 搜索路径劫持  
- **攻击场景**：代码执行 `sys.path.append("../../")`，将当前工作目录的上级目录加入 Python 模块搜索路径。若攻击者能在该上级目录（如 `../`）中植入恶意 Python 文件（如 `amia.py` 或 `models.py`），则后续 `import` 语句将加载恶意模块，实现任意代码执行。  
- **影响范围**：完整系统沦陷，攻击者可通过恶意模块执行任意命令、窃取数据、持久化后门等。  
- **已有防护**：无。该操作为硬编码，且未校验目标目录的权限或内容。

---

#### **中风险**

**3. 输入数据注入（提示注入）**  
- **漏洞类型**：输入注入（Prompt Injection）  
- **攻击场景**：数据文件中的 `"text"` 字段未经任何过滤直接作为模型提示（`prompt`）传入 `vlm.generate()` 或 `defender.defend()`。若数据来源被篡改，攻击者可构造恶意提示，绕过模型的安全对齐（如越狱指令），诱导模型输出有害内容（如仇恨言论、隐私数据等）。  
- **影响范围**：模型生成结果失控，导致输出不安全内容，影响下游应用（如内容审核系统）的可靠性。但该风险不直接危害系统环境，属应用层安全缺陷。  
- **已有防护**：`USE_AMIA` 启用时调用 `AMIA` 防御，该防御旨在对抗越狱攻击，但本质是算法层缓解而非输入过滤，且防御效果依赖于其内部实现，可能并非完全有效。

**4. 资源耗尽（拒绝服务）**  
- **漏洞类型**：资源耗尽（Uncontrolled Resource Consumption）  
- **攻击场景**：攻击者通过提供超大尺寸图片（如 10000×10000 像素）或超长文本（如数万 token）的数据条目，导致 `Image.open()` 消耗大量内存，或模型推理（`vlm.generate`）占用 GPU 显存直至溢出，造成进程崩溃或系统无响应。  
- **影响范围**：测试服务可用性受损，可能中断正在进行的批量测试，影响任务完整性。  
- **已有防护**：无。代码未对图片尺寸、文件大小、提示长度做任何限制。

**5. 异常处理不完整（拒绝服务/崩溃）**  
- **漏洞类型**：未捕获的异常（Improper Exception Handling）  
- **攻击场景**：代码仅对图片加载失败做了 `try-except` 并跳过，但未捕获模型推理（`vlm.generate` 或 `defender.defend`）、文件写入、JSON 序列化等步骤可能抛出的异常（如 CUDA 错误、内存不足、编码错误）。攻击者可通过构造特殊输入触发这些异常，使程序提前终止，中断整个测试流程。  
- **影响范围**：测试任务中断，无法完成既定批量处理，可能导致结果缺失或重复劳动。  
- **已有防护**：对图片加载有部分异常处理，但不覆盖主要业务逻辑，防护不完整。

---

#### **低风险**

**6. 日志信息泄露**  
- **漏洞类型**：敏感信息记录（Information Exposure in Logs）  
- **攻击场景**：程序在控制台和输出文件 `results/defense_results_v3.jsonl` 中记录完整的模型输入（`query`）、输出（`output`）、分析（`analysis`）和图片路径。若日志被未授权访问（如存储权限配置不当或共享目录），可能泄露测试数据中的敏感内容（如用户隐私、商业机密）或模型内部分析细节。  
- **影响范围**：数据保密性受损，违反数据保护合规要求（如 GDPR）。  
- **已有防护**：无。输出内容未脱敏或过滤。

**7. 不安全文件写入（符号链接攻击）**  
- **漏洞类型**：不安全文件操作（Unsafe File Write）  
- **攻击场景**：结果文件路径 `results/defense_results_v3.jsonl` 固定，且以追加模式写入。若 `results` 目录被攻击者通过符号链接指向敏感系统文件（如 `/etc/shadow`），且运行进程具有写入权限，则可能覆盖或损坏该文件。  
- **影响范围**：系统文件或其它重要数据被破坏，但前提是攻击者已拥有目录写权限，故风险相对较低。  
- **已有防护**：使用 `os.makedirs` 创建目录，但未检查路径是否为符号链接或是否有足够权限。

**8. 调试代码残留**  
- **漏洞类型**：配置不当（Debug Code Residual）  
- **攻击场景**：注释中保留 `debug_idx` 调试开关，若未来开发者误启用（取消注释），将只处理单个样本，可能导致测试结果不完整且难以察觉，影响最终评估结论的可靠性。  
- **影响范围**：测试结果失真，可能隐藏模型的实际安全性能问题。  
- **已有防护**：无，注释代码未清理。

---

#### 总结建议

- 立即修复 **高风险** 的路径遍历和模块搜索路径污染问题，例如使用白名单限制图片目录、移除 `sys.path.append("../../")` 改用绝对路径或项目相对路径。
- 对 **中风险** 引入输入大小限制、增强异常捕获、考虑对 `prompt` 进行基本过滤（如长度限制、特殊字符转义）作为辅助防护。
- 对 **低风险** 进行日志脱敏、文件写入权限校验、移除调试代码。

以上分析仅针对框架代码本身的安全性问题，不涉及 AMIA 或 LLaVA 模型内部的算法安全。

### 安全风险分析报告

> **模块：** `成员代码/fengyongjia/watermarkLSB.py`  
> **分析时间：** 2026-06-27

经过对提供的改进LSB图像隐写系统代码进行审查，发现以下安全风险。按严重程度（高/中/低）排序，每个风险均包含漏洞类型、攻击场景描述、影响范围，并对已有但不完整的防护措施进行说明。

#### **中风险**

**1. 路径硬编码与路径穿越漏洞（R-01）**  
- **漏洞类型**：路径硬编码（Hardcoded Path）+ 路径遍历（Path Traversal）  
- **攻击场景**：代码中 `Image.open('buptgray.bmp')` 和 `stego_img.save('buptgraystego1.bmp')` 使用硬编码相对路径。若将此函数暴露给外部调用（如 Web 服务或命令行工具），攻击者传入类似 `../../../etc/passwd` 的路径，可通过路径穿越读取/写入系统任意位置文件。此外，`cv2.imread()` 同样接收未校验的路径字符串。  
- **影响范围**：任意文件读取和写入，可泄露系统敏感文件或在系统目录写入恶意文件。  
- **已有防护**：无。所有文件路径均为硬编码字符串或直接传入底层库函数，未做任何规范化或边界校验。

**2. 输入消息未校验（R-02）**  
- **漏洞类型**：输入验证缺失（Missing Input Validation）  
- **攻击场景**：`hide_message_improved_sequential()` 的 `secret_msg` 参数未进行长度限制和内容校验。攻击者可传入空字符串（导致索引错误）、超长字符串（超出图像容量导致异常），或包含不可打印字符/控制字符的消息（影响日志和显示系统）。  
- **影响范围**：程序崩溃、资源浪费、日志注入风险。  
- **已有防护**：仅检查消息长度是否超过图像容量（`msg_length > height * width`），但未设置合理上限、未校验空消息、未过滤特殊字符。

**3. 文件格式与大小未校验 — 资源耗尽风险（R-03）**  
- **漏洞类型**：资源耗尽（Uncontrolled Resource Consumption）  
- **攻击场景**：`Image.open()` 和 `cv2.imread()` 未校验输入文件是否为合法图像格式及文件大小。攻击者可提供超大尺寸图片（如 20000×20000 像素）或伪装成图片的非图像文件，导致内存耗尽（OOM）或程序长时间无响应。  
- **影响范围**：系统内存耗尽，进程崩溃，可能影响同主机上其他服务。  
- **已有防护**：无。代码未对图片文件大小、像素尺寸、格式做任何检查。

---

#### **低风险**

**4. 异常处理不完整（R-04）**  
- **漏洞类型**：不当异常处理（Improper Exception Handling）  
- **攻击场景**：代码仅在最外层 `if __name__ == "__main__"` 处使用裸 `except Exception as e` 捕获所有异常。若内部函数（如 `Image.open()`、`cv2.imread()`、文件 I/O）抛出具体异常（`FileNotFoundError`、`PermissionError`、`PIL.UnidentifiedImageError` 等），无法区分处理，错误信息仅打印 `f"执行出错: {e}"`，不利于安全审计和问题溯源。  
- **影响范围**：安全事件难以追踪，异常恢复机制缺失。  
- **已有防护**：部分——至少在外层有 try-except，未让程序静默崩溃。

**5. 固定随机种子安全隐患（R-05）**  
- **漏洞类型**：弱随机性（Weak Randomness / Predictable Seeding）  
- **攻击场景**：代码使用 `np.random.seed(2021)` 固定种子生成嵌入位置。虽然这保证了可提取性（嵌入端和提取端使用相同种子），但种子值硬编码在源码中，任何获得源码的人都能完全确定嵌入位置序列，使隐写内容被精确定位和提取，失去了"隐蔽"的安全意义。  
- **影响范围**：隐写内容的机密性完全依赖源码保密，而非密钥控制。一旦源码泄露，所有使用此代码嵌入的秘密消息均可被提取。  
- **已有防护**：无。种子硬编码于源码中。

**6. 输出文件覆盖风险（R-06）**  
- **漏洞类型**：不安全文件操作（Unsafe File Write）  
- **攻击场景**：输出文件路径 `'buptgraystego1.bmp'` 硬编码，多次运行会直接覆盖之前的输出文件，无任何提示或备份。若未来代码改为接收用户指定的输出路径，也可能通过路径遍历覆盖系统关键文件。  
- **影响范围**：数据丢失（覆盖已有实验结果），文件完整性被破坏。  
- **已有防护**：无。

---

#### 总结建议

- 优先修复 **中风险** 的路径硬编码/路径穿越和输入验证缺失问题，使用 `os.path.realpath()` 规范化路径、添加消息长度和内容校验。
- 对 **低风险** 进行文件格式/大小限制、增强异常捕获粒度、引入基于密钥的随机种子（而非硬编码固定值）、添加输出文件覆盖保护。

**风险名称：** 输入文件大小与格式未校验

**攻击场景：** 攻击者将超大文件（如 1GB .bmp）重命名为 `bupt.bmp`，导致程序内存耗尽（拒绝服务）；或将非 BMP 文件传入导致 cv2.imread() 返回 None 引发未处理异常。
**✅ 修复状态：** 已修复 — 新增 validate_image_file()，校验扩展名白名单（.bmp）和文件大小上限（50MB）

### 风险 4：异常处理不完整 ⚠️ 低危

**风险编号：** R-04
**风险名称：** 异常处理不完整
**严重程度：** 低危（CVSS 3.1）
**攻击场景：** cv2.dct()、cv2.imwrite()、np.save() 等操作未做细粒度异常捕获，程序异常终止时安全事件难以溯源。
**✅ 修复状态：** 已修复 — 使用具体异常类型（FileNotFoundError, ValueError, OSError, AssertionError, cv2.error）替换裸异常，区分安全事件并给出明确错误提示
